Learn HCL (HashiCorp Configuration Language), understand providers and modules, master state management, and build production-grade infrastructure that scales. Whether you're managing one VPS or ten thousand servers, Terraform gives you the power to control it all.

// Why Terraform

Terraform was created by Mitchell Hashimoto in 2014 and has become the industry standard for Infrastructure as Code. Unlike cloud-specific tools, Terraform is provider-agnostic—you can manage AWS, GCP, Azure, Kubernetes, and on-premises infrastructure with the same language and workflow.

The declarative approach means you describe your desired state, and Terraform figures out how to achieve it. This makes infrastructure reproducible, versionable, and auditable.

Infrastructure that scales. Code that lasts.

What is Infrastructure as Code?

Infrastructure as Code (IaC) is the practice of managing and provisioning computer data centers through machine-readable definition files, rather than physical hardware configuration or interactive configuration tools.

The Declarative Paradigm

Terraform is declarative—you describe what you want, not how to get there:

  • Imperative: "Create a VPC, then create a subnet, then launch an EC2..."
  • Declarative: "I want an EC2 instance in a VPC with these specifications."

The Terraform Workflow

  1. Write: Create configuration files describing your infrastructure
  2. Init: Run terraform init to download providers
  3. Plan: Run terraform plan to preview changes
  4. Apply: Run terraform apply to make changes
  5. Destroy: Run terraform destroy to tear down

Installing Terraform

# On Linux/macOS
curl -fsSL https://get.terraform.io | sh

# On macOS with Homebrew
brew install terraform

# Verify
terraform version

Your First Configuration

# main.tf
resource "local_file" "hello" {
  filename = "hello.txt"
  content  = "Hello from Terraform!"
}

# Run the workflow
terraform init
terraform plan
terraform apply

HashiCorp Configuration Language

HCL is Terraform's domain-specific language. It's designed to be both human-readable and machine-parseable.

Blocks and Arguments

# This is a block
resource "aws_instance" "web" {
  # These are arguments
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
  
  tags = {
    Name = "HelloWorld"
  }
}

Data Types

# Strings
name = "web-server"

# Numbers
count = 5

# Booleans
enable_monitoring = true

# Lists
ports = [80, 443, 8080]

# Maps
tags = {
  Name = "web-server"
  Environment = "production"
}

References

resource "aws_security_group" "web" {
  name = "web-sg"
}

resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  vpc_security_group_ids = [aws_security_group.web.id]
}

Expressions

# String interpolation
name = "server-${var.environment}"

# Conditionals
instance_type = var.is_production ? "t3.large" : "t3.micro"

# Functions
upper_name = upper(var.name)

Data Sources

Data sources let you fetch information from existing infrastructure:

data "aws_ami" "ubuntu" {
  most_recent = true
  owners      = ["099720109477"]
  
  filter {
    name   = "name"
    values = ["ubuntu/images/hvm-ssd/ubuntu-jammy-22.04-*"]
  }
}

resource "aws_instance" "web" {
  ami = data.aws_ami.ubuntu.id
}

Input Variables

Variables make configurations reusable:

  type        = string
  default     = "t2.micro"

Using Variables

resource "aws_instance" "web" {
  instance_type = var.instance_type
  # ...
}

Passing Variables

# Via command line
terraform apply -var="environment=prod"

# Via file
terraform apply -var-file="prod.tfvars"

# Via environment variable
export TF_VAR_environment=prod

output "instance_id" {
  description = "The ID of the EC2 instance"
  value       = aws_instance.web.id
}

output "instance_public_ip" {

What is State?

Terraform uses state to map your configuration to real-world resources. It stores resource identifiers and attributes.

Local State

By default, Terraform stores state in terraform.tfstate:

  • Works for individual development
  • No collaboration support
  • Risk of corruption

Remote Backends

terraform {
  backend "s3" {
    bucket         = "my-terraform-state"
    key            = "prod/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-locks"
  }
}

State Commands

# List resources
terraform state list

# Show resource
terraform state show aws_instance.web

# Move resource
terraform state mv aws_instance.web aws_instance.app

# Remove from state
terraform state rm aws_instance.old

Why Modules?

Modules group related resources and enable reuse:

  • Organize code by component
  • Create reusable infrastructure
  • Version control infrastructure

Creating a Module

# modules/ec2/main.tf
variable "instance_type" {
  type    = string
  default = "t2.micro"
}

resource "aws_instance" "this" {
  instance_type = var.instance_type
}

output "instance_id" {
  value = aws_instance.this.id
}

Using a Module

module "web_server" {
  source = "./modules/ec2"
  
  instance_type = "t3.small"
}

Module Sources

# From Terraform Registry
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "3.0.0"
}

# From GitHub
module "example" {
  source = "github.com/user/repo//path/to/module"
}

What are Workspaces?

Workspaces let you manage multiple environments with a single configuration. Each workspace has its own state.

Managing Workspaces

# List workspaces
terraform workspace list

# Create workspace
terraform workspace new staging

# Switch workspace
terraform workspace select production

# Delete workspace
terraform workspace delete dev

Workspace-Specific Configuration

locals {
  environment = terraform.workspace
  
  configs = {
    dev = { instance_type = "t3.micro" }
    staging = { instance_type = "t3.small" }
    production = { instance_type = "t3.large" }
  }
}

resource "aws_instance" "web" {
  instance_type = local.configs[local.environment].instance_type
}

When to Use Workspaces

  • Development vs production
  • Feature branches
  • Short-lived environments

What are Provisioners?

Provisioners let you execute commands on created resources for bootstrapping.

local-exec

Runs commands on the machine running Terraform:

resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
  
  provisioner "local-exec" {
    command = "echo 'Instance ${self.public_ip} created!'"
  }
}

remote-exec

Runs commands on the created resource via SSH:

resource "aws_instance" "web" {
  # ...
  
  connection {
    type        = "ssh"
    host        = self.public_ip
    user        = "ubuntu"
    private_key = file("~/.ssh/id_rsa")
  }
  
  provisioner "remote-exec" {
    inline = [
      "sudo apt-get update",
      "sudo apt-get install -y nginx"
    ]
  }
}

Cloud-Init

Better alternative for cloud instances:

resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
  
  user_data = <<-EOF
              #!/bin/bash
              apt-get update
              apt-get install -y nginx
              EOF
}

Automated Testing with Terratest

package test

import (
    "testing"
    "github.com/gruntwork-io/terratest/modules/terraform"
    "github.com/stretchr/testify/assert"
)

func TestTerraformWebServer(t *testing.T) {
    terraformOptions := &terraform.Options{
        TerraformDir: "../examples/web-server",
    }
    
    defer terraform.Destroy(t, terraformOptions)
    terraform.InitAndApply(t, terraformOptions)
    
    instanceID := terraform.Output(t, terraformOptions, "instance_id")
    assert.NotEmpty(t, instanceID)
}

GitHub Actions

name: Terraform

on:
  push:
    branches: [main]
  pull_request:

jobs:
  terraform:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - uses: hashicorp/setup-terraform@v2
      
      - name: Terraform Init
        run: terraform init
      
      - name: Terraform Plan
        run: terraform plan
      
      - name: Terraform Apply
        if: github.ref == 'refs/heads/main'
        run: terraform apply -auto-approve

Policy Enforcement

Sentinel policies enforce governance:

# Example: Require tags on all resources
import "tfplan/v2" as tfplan

main = rule {
  all tfplan.resource_changes as _, rc {
    all rc.change.after as key, value {
      key == "tags" or key == "Name"
    }
  }
}

Dynamic Blocks

Create multiple similar blocks programmatically:

resource "aws_security_group" "web" {
  name = "web-sg"
  
  dynamic "ingress" {
    for_each = var.ports
    content {
      from_port   = ingress.value
      to_port     = ingress.value
      protocol    = "tcp"
      cidr_blocks = ["0.0.0.0/0"]
    }
  }
}

for_each

Create multiple resources from a map or set:

resource "aws_instance" "server" {
  for_each = {
    web = { ami = "ami-1", type = "t3.small" }
    api = { ami = "ami-2", type = "t3.medium" }
  }
  
  ami           = each.value.ami
  instance_type = each.value.type
  tags          = { Name = each.key }
}

Splat Expressions

# Get all instance IDs
instance_ids = aws_instance.web[*].id

# Use in for_each
for_each = toset(aws_instance.server[*].id)
